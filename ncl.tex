\begin{abstract}
\noindent \textbf{Abstract.} This research presents the \textit{Nepal Compliance Ledger (NCL)}, a regulatory technology prototype for supervising high-frequency financial flows before settlement. The system combines an \textbf{Agent-Based Model (ABM)}, a sliding-window velocity detector, bounded graph-cycle detection, and a permissioned ledger audit layer with Merkle roots, hash-linked blocks, policy binding, and simulated validator quorum signatures. The upgraded implementation simulates $N=10{,}000$ autonomous actors and processes $100{,}000$ transactions. The run produced $38{,}566$ true positives, $39$ false positives, $6{,}601$ false negatives, and $54{,}794$ true negatives, achieving a detection precision of \textbf{99.90\%}, recall of \textbf{85.39\%}, and an F1 score of \textbf{0.9207}. The ledger sealed $196$ auditable blocks with intact chain verification, showing how algorithmic supervision and tamper-evident audit infrastructure can be combined in a reproducible AML compliance simulation.
\end{abstract}

\section{Introduction}
AML supervision often suffers from settlement latency: suspicious flows may only be discovered after money has already moved across institutions. In Nepal and similar financial environments, \textit{structuring} or smurfing remains a practical evasion strategy, where a larger illicit transfer is fragmented into many smaller payments below scalar reporting thresholds.

This research treats the monitoring problem as a combination of velocity, graph topology, and auditability. Instead of asking only whether a single transaction exceeds a limit, NCL evaluates whether an actor is transacting too frequently inside a rolling time window and whether a new transfer closes a short circular path. The ledger layer records compliance decisions in a hash-chained permissioned structure so participating banks and regulators can verify the audit trail.

The velocity of funds can be represented as:
\begin{equation}
    V = \frac{\sum_{i=1}^{n} \Delta \text{Amount}_i}{\Delta \text{Time}}
\end{equation}
The core hypothesis is that a permissioned compliance environment can enforce velocity and topology constraints before settlement, reducing the delay between suspicious behavior and supervisory response.

\section{Methodology}

\subsection{Architectural Simulation}
The Hub-and-Spoke simulation environment models the transactional logic of a permissioned banking network. It consists of four operational layers:
\begin{enumerate}
    \item \textbf{User Interface Layer:} Simulates mobile banking transaction requests.
    \item \textbf{Compliance Brain:} Runs identity validation, velocity checks, and graph-cycle detection.
    \item \textbf{Immutable Ledger Layer:} Records clean and flagged decisions in a hash-linked audit structure.
    \item \textbf{Regulator Output Layer:} Provides a real-time supervisory feed for audit review.
\end{enumerate}

\subsection{Algorithmic Logic 1: Sliding Window Protocol}
To address the weakness of fixed-window counters, the NCL architecture implements a \textit{sliding-window log}. For each sender, the engine keeps a queue of recent transaction timestamps and prunes values outside the policy window.
\begin{equation}
    \text{Count}(a, T_{now}) = \sum_{i=1}^{n} \mathbb{I}(T_{now} - t_i < W)
\end{equation}
where $\mathbb{I}$ is the indicator function, $a$ is the sender, and $W=300\,\text{s}$ is the window size. A transaction is flagged when the sender's count reaches the threshold $\tau=5$ within the active window.

\subsection{Algorithmic Logic 2: Graph Cycle Detection}
Scalar AML checks cannot reliably detect \textit{layering}, where funds are routed through multiple accounts to obscure origin. NCL models transactions as a directed graph $G=(V,E)$. For a proposed transaction $e(u,v)$, the system checks whether a short path already exists from $v$ back to $u$:
\begin{equation}
    v \rightarrow \dots \rightarrow u \quad \text{where} \quad |P| \le k
\end{equation}
If funds return to the origin node $u$ within $k=3$ hops, the topology is flagged as a circular trading ring. The implementation also uses a $900\,\text{s}$ graph memory window so the detector focuses on recent layering behavior.

\setcounter{algorithm}{1}
\begin{algorithm}[H]
\caption{Recursive Cycle Detection (DFS)}
\begin{algorithmic}[1]
\Function{DetectCycle}{$origin, current, depth$}
    \If{$current == origin$ \textbf{AND} $depth > 0$}
        \State \Return \textbf{TRUE} \Comment{Cycle found}
    \EndIf
    \If{$depth \ge MAX\_DEPTH$}
        \State \Return \textbf{FALSE}
    \EndIf
    \For{each $neighbor$ in $AdjacencyList[current]$}
        \If{\Call{DetectCycle}{$origin, neighbor, depth+1$}}
            \State \Return \textbf{TRUE}
        \EndIf
    \EndFor
    \State \Return \textbf{FALSE}
\EndFunction
\end{algorithmic}
\end{algorithm}

\subsection{Algorithmic Logic 3: Permissioned Ledger Verification}
The advanced NCL prototype upgrades the earlier append-only log into a permissioned block ledger. Every compliance decision is converted into a canonical audit record and batched into blocks of $512$ records. Each block header contains the previous block hash, Merkle root, policy hash, proposer node, transaction count, and validator signatures.

For block $B_j$, the chain commitment is:
\begin{equation}
    H(B_j) = SHA256(index_j \parallel prevHash_j \parallel merkleRoot_j \parallel policyHash_j)
\end{equation}
The verifier accepts a run only when all previous-hash links, Merkle roots, block hashes, and simulated validator signatures satisfy the policy quorum of $4$ out of $5$ validators.

\section{Discussion}
\subsection{Architectural Justification: Why a Permissioned Ledger?}
A centralized database can record transaction decisions, but it also concentrates trust in a single administrator or institution. NCL uses a permissioned ledger model because AML supervision is multi-institutional: banks and regulators need a shared audit object whose integrity can be independently verified. In the prototype, each block binds transaction decisions to a previous hash, Merkle root, policy hash, and validator quorum, making later manipulation detectable.

\subsection{Detection Behavior}
Most detected malicious activity was captured by the sliding-window velocity protocol, which is expected because structuring is represented as repeated low-value transfers. The graph-cycle detector contributes an additional layer for circular trading rings. This two-part approach keeps the policy interpretable: investigators can see whether a transaction was flagged because of excessive velocity or because it completed a suspicious path.

\subsection{Limitations \& Future Work}
The current implementation is a deterministic simulation, not a production banking system. Validator signatures are simulated rather than backed by real institutional keys or HSMs. The ABM uses synthetic behavior distributions that would require calibration against legally shareable banking data. Future work should evaluate stochastic windows, richer graph anomaly scoring, and privacy-preserving regulatory data sharing.

\section{Conclusion}
The Nepal Compliance Ledger simulation provides empirical evidence that algorithmic AML supervision can be modeled as a pre-settlement compliance problem. By combining sliding-window velocity checks, graph-cycle detection, and a permissioned audit ledger, NCL demonstrates how high-frequency financial flows can be filtered and recorded in a tamper-evident system. The upgraded 100k-transaction run shows strong precision, reproducible ledger integrity, and a clear path for expanding the prototype into larger regulatory technology experiments.
